# NanoClaw Role 2: Intelligent Test Executor

**Runtime Auditor & Health Monitor for NanoClaw Test Scripts**

---

## What is Role 2?

Role 2 is the execution engine that takes test scripts generated by Role 1 (Test Generator) and runs them in an isolated sandbox with comprehensive health monitoring. Unlike standard test runners, Role 2 acts as a **runtime auditor**, watching not just whether tests pass, but whether the application under test remains healthy during execution.

**Key Differentiator:**
- **Standard Runner**: Test passes ✓
- **Role 2**: Test passes ✓ BUT browser memory leaked 500MB ✗ → FAIL

## Core Components

### 1. Executor Service (`executor/`)
Runs pytest with Playwright in isolated Docker containers.

**Features:**
- Fresh browser context per test (no cache, no cookies)
- `pytest --json-report --trace` for machine-readable results
- Real-time health monitoring integration
- Sequential execution with error handling

### 2. Monitor Sidecar (`monitor/`)
Watches browser vitals during test execution.

**Metrics Collected:**
| Metric | What It Detects | Threshold |
|--------|----------------|-----------|
| Memory Heap | Sawtooth patterns = memory leak | Growing > 100MB in 60s |
| CPU Usage | Bad JS loops, infinite recursion | >80% during idle |
| Network | Background API failures | Any 4xx/5xx |
| Console | Uncaught errors, warnings | Any stack trace |

**Technology:**
- `psutil` for process metrics
- Chrome DevTools Protocol (CDP) for browser internals
- Streaming metrics via WebSocket to executor

### 3. Jira Integrator (`jira_integrator/`)
Updates Jira tickets with structured test results.

**Actions:**
- Extracts Jira ticket ID from test metadata
- Posts comment with:
  - Status (PASS/FAIL)
  - Health Grade (HEALTHY/WARNING/CRITICAL)
  - Metrics summary
- Attaches evidence package:
  - `metrics.csv` - Peak memory, CPU, network errors
  - `trace.zip` - Playwright trace viewer file
  - `logs.txt` - Merged terminal + console output

### 4. Evidence Packager
Builds forensic evidence folder for each test run.

**Contents:**
```
results/{job_id}/
├── metrics.csv          # Time-series health data
├── logs.txt            # Terminal + console merged
├── trace.zip           # Playwright trace
├── screenshot/         # Failure screenshots
└── health_report.json  # Overall health grade
```

### 5. WhatsApp Interface
Trigger tests via WhatsApp command.

**Commands:**
- `/run {job_id}` - Execute specific test
- `/status {job_id}` - Check execution status
- `/results {job_id}` - Get evidence package link

### 6. MS Teams Integration
Trigger tests and monitor results directly in Microsoft Teams.

**Commands:**
- `/run {job_id}` - Execute specific test
- `/status {job_id}` - Check execution status
- `/results {job_id}` - Get evidence package link
- `/list` - List all executions
- `/help` - Show help message

**Features:**
- Rich Adaptive Cards with health grade badges (HEALTHY/WARNING/CRITICAL)
- Real-time status updates with progress indicators
- Evidence package sharing with Jira integration
- Support for both channel conversations and 1:1 chats

See [Teams Integration Guide](docs/teams-integration.md) for detailed setup instructions.

## Health Grading System

Role 2 assigns a health grade based on observed metrics during test execution:

### HEALTHY
- Test passes ✓
- Memory stable (no sawtooth patterns)
- CPU < 50% during idle
- Zero console errors
- Zero network errors

### WARNING
- Test passes ✓ BUT:
  - Minor console warnings (non-blocking)
  - Slow API responses (>2s but <5s)
  - Memory growth < 50MB over test duration
  - Occasional CPU spikes (60-80%)

### CRITICAL
- Test fails ✗ OR:
  - Memory leak detected (sawtooth pattern, >100MB growth)
  - CPU sustained >80% during idle
  - Uncaught exceptions in console
  - Network errors (4xx/5xx)
  - Browser freeze detected

## Monitoring Architecture

### The Sidecar Pattern

Role 2 uses a **sidecar monitoring approach** rather than running monitoring inside the executor container.

**Why?** Running monitoring inside the test container only sees container limits, not actual browser pain. The host Mac could be screaming while the container looks calm.

```
┌─────────────────────────────────────────────────┐
│  Docker Host (Your Mac)                         │
│  ┌──────────────────┐  ┌──────────────────┐    │
│  │  Executor Cont   │  │  Monitor Cont    │    │
│  │  ┌────────────┐  │  │  ┌────────────┐  │    │
│  │  │ Playwright │  │  │  │ CDP Client │◄─┼────┼── Chrome DevTools
│  │  │   Test     │  │  │  │ psutil     │  │    │   Protocol
│  │  └────────────┘  │  │  └────────────┘  │    │
│  └──────────────────┘  └──────────────────┘    │
│         ▲                     │                │
│         │                     │                │
│         └─────── WebSocket ────┘                │
│            (real-time metrics)                 │
└─────────────────────────────────────────────────┘
```

### Monitoring Technologies

| Technology | Purpose |
|------------|---------|
| **Chrome DevTools Protocol (CDP)** | Direct browser internals access (memory heap, console, network) |
| **psutil** | Process-level CPU and memory metrics |
| **pytest-json-report** | Machine-readable test results |
| **Playwright Tracing** | Video-like execution replay |

## Test Execution Flow

```
1. TRIGGER (WhatsApp or API)
   ↓
2. PULL SCRIPT from shared volume
   ↓
3. SPAWN fresh browser context
   ↓
4. START monitoring sidecar
   ↓
5. RUN pytest --json-report --trace
   ↓ (concurrent monitoring)
6. COLLECT metrics from sidecar
   ↓
7. GENERATE evidence package
   ↓
8. CALCULATE health grade
   ↓
9. UPDATE Jira ticket
   ↓
10. NOTIFY via WhatsApp
```

## API Endpoints

### Executor API (port 8001)

**POST /api/v1/execute**
Execute a test script.

```json
{
  "job_id": "abc-123-def",
  "jira_ticket": "QA-456",
  "script_path": "/shared/scripts/test_login.py"
}
```

**Response:**
```json
{
  "execution_id": "exec-uuid",
  "status": "running",
  "trace_url": "http://localhost:8001/trace/exec-uuid"
}
```

**GET /api/v1/status/{execution_id}**
Get execution status.

**Response:**
```json
{
  "execution_id": "exec-uuid",
  "status": "completed",
  "test_result": "PASS",
  "health_grade": "WARNING",
  "metrics": {
    "peak_memory_mb": 450,
    "peak_cpu_percent": 75,
    "network_errors": 0,
    "console_errors": 2
  },
  "evidence_package": "/shared/results/exec-123/"
}
```

**GET /api/v1/trace/{execution_id}**
Serve Playwright trace viewer.

### Monitor API (port 8002)

**WS /ws/metrics**
WebSocket endpoint for real-time metric streaming.

**Message Format:**
```json
{
  "timestamp": "2026-03-08T15:30:00Z",
  "memory_heap_mb": 234.5,
  "cpu_percent": 45.2,
  "network_errors": [],
  "console_errors": []
}
```

```python
    # Test implementation
```
